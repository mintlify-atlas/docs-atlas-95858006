title: "Quickstart: write your first redb database"
description: "Go from zero to a working redb database in minutes: add the dependency, define a table, write data, read it back, and iterate with a range query."
---

This guide walks you through every step needed to write and read data with redb. By the end you will have a working Rust program that opens a database, inserts records in a write transaction, reads them back in a read transaction, and iterates over a range — using real, runnable code.

<Steps>

<Step title="Add the dependency">

Add redb to your `Cargo.toml`:

```toml Cargo.toml
[dependencies]
redb = "4"
```

Then fetch it:

```bash
cargo fetch
```

<Note>
redb requires Rust 1.89 or later. Run `rustup update stable` if you are on an older toolchain.
</Note>

</Step>

<Step title="Define a table">

Table definitions are typed constants declared at the module level. The two type parameters are the key type and value type. redb checks these at compile time — opening a table with the wrong types is a compile error.

```rust main.rs
use redb::TableDefinition;

// A table mapping &str keys to u64 values, named "my_data"
const TABLE: TableDefinition<&str, u64> = TableDefinition::new("my_data");
```

You can also use integer keys directly:

```rust int_keys.rs
use redb::TableDefinition;

// A table mapping u64 keys to u64 values
const TABLE: TableDefinition<u64, u64> = TableDefinition::new("my_data");
```

</Step>

<Step title="Open a database">

`Database::create` opens an existing file or creates a new one. The path can be any type that implements `AsRef<Path>`.

```rust main.rs
use redb::{Database, Error, ReadableDatabase, TableDefinition};

const TABLE: TableDefinition<&str, u64> = TableDefinition::new("my_data");

fn main() -> Result<(), Error> {
    let db = Database::create("my_db.redb")?;

    // ...
    Ok(())
}
```

<Tip>
Use `Database::open` instead of `Database::create` when you want to fail if the file does not already exist.
</Tip>

</Step>

<Step title="Write data">

All writes happen inside a `WriteTransaction`. Open a table, insert key-value pairs, then commit. If the transaction is dropped without calling `commit`, all changes are automatically rolled back.

```rust main.rs
use redb::{Database, Error, ReadableDatabase, TableDefinition};

const TABLE: TableDefinition<&str, u64> = TableDefinition::new("my_data");

fn main() -> Result<(), Error> {
    let db = Database::create("my_db.redb")?;

    let write_txn = db.begin_write()?;
    {
        let mut table = write_txn.open_table(TABLE)?;
        table.insert("alice", &42)?;
        table.insert("bob", &100)?;
        table.insert("carol", &7)?;
    }
    write_txn.commit()?;

    Ok(())
}
```

The inner block ensures `table` is dropped before `commit` is called — redb requires all table handles to be released before a transaction can be committed.

<Warning>
If your process crashes between `begin_write` and `commit`, redb discards the partial transaction automatically on the next open. No manual recovery is needed.
</Warning>

</Step>

<Step title="Read data">

Read transactions are opened with `begin_read`. Multiple read transactions can run concurrently with a write transaction without blocking each other.

```rust main.rs
use redb::{Database, Error, ReadableDatabase, ReadableTable, TableDefinition};

const TABLE: TableDefinition<&str, u64> = TableDefinition::new("my_data");

fn main() -> Result<(), Error> {
    let db = Database::create("my_db.redb")?;

    // (insert records as shown in the previous step)

    let read_txn = db.begin_read()?;
    let table = read_txn.open_table(TABLE)?;

    // get() returns Option<AccessGuard<V>>, call .value() to get the inner value
    if let Some(value) = table.get("alice")? {
        println!("alice: {}", value.value()); // prints: alice: 42
    }

    // Missing keys return None
    assert!(table.get("zoe")?.is_none());

    Ok(())
}
```

`get` returns `Ok(None)` for keys that do not exist, so you never need to distinguish between a missing key and a storage error without explicitly checking.

</Step>

<Step title="Iterate with a range query">

Use `range` to iterate over a contiguous slice of keys. The range argument accepts any `RangeBounds<K>`.

```rust main.rs
use redb::{Database, Error, ReadableDatabase, ReadableTable, TableDefinition};

const TABLE: TableDefinition<&str, u64> = TableDefinition::new("my_data");

fn main() -> Result<(), Error> {
    let db = Database::create("my_db.redb")?;

    let write_txn = db.begin_write()?;
    {
        let mut table = write_txn.open_table(TABLE)?;
        table.insert("alice", &42)?;
        table.insert("bob", &100)?;
        table.insert("carol", &7)?;
    }
    write_txn.commit()?;

    let read_txn = db.begin_read()?;
    let table = read_txn.open_table(TABLE)?;

    // Iterate all entries — keys are returned in sorted order
    for result in table.range::<&str>(..)? {
        let (key, value) = result?;
        println!("{}: {}", key.value(), value.value());
    }
    // Output (alphabetical):
    //   alice: 42
    //   bob: 100
    //   carol: 7

    // Bounded range: keys from "alice" up to (but not including) "carol"
    for result in table.range("alice".."carol")? {
        let (key, value) = result?;
        println!("{}: {}", key.value(), value.value());
    }

    Ok(())
}
```

<Tip>
Keys in redb are always stored and iterated in ascending sorted order, matching `BTreeMap` semantics. String keys sort lexicographically; integer keys sort numerically.
</Tip>

</Step>

</Steps>

## Error handling

redb uses the `Error` type as a catch-all, or you can match the more specific sub-types for finer control:

```rust error_handling.rs
use redb::{Database, DatabaseError, Error, TableError, TableDefinition};

const TABLE: TableDefinition<&str, u64> = TableDefinition::new("my_data");

fn open_db(path: &str) -> Result<Database, DatabaseError> {
    // Database::open fails with DatabaseError::Io if the file doesn't exist
    Database::open(path)
}

fn read_value(db: &Database, key: &str) -> Result<Option<u64>, Error> {
    let read_txn = db.begin_read()?;
    let table = match read_txn.open_table(TABLE) {
        Ok(t) => t,
        Err(TableError::TableDoesNotExist(_)) => {
            // Table hasn't been written to yet — treat as empty
            return Ok(None);
        }
        Err(e) => return Err(e.into()),
    };
    Ok(table.get(key)?.map(|v| v.value()))
}
```

<Note>
For most applications, propagating `redb::Error` with `?` is sufficient. Match on sub-types only when you need to recover from specific conditions like a missing table or a corrupted file.
</Note>
